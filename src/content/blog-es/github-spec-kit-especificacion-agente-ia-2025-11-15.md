# GitHub spec-kit: tres comandos para que su agente IA construya lo que usted quiere realmente

Desde mayo de 2025, BMAD ha demostrado que es posible salir del vibe coding. Al imponer un equipo de agentes especializados —analista, PM, arquitecto, dev, QA— que se transfieren artefactos estructurados, garantiza que la IA produce lo que usted realmente ha especificado. Funciona. También es un framework completo, con una curva de configuración real.

GitHub acaba de responder con el **GitHub spec-kit**, lanzado en noviembre de 2025: ¿y si pudiéramos hacer la misma apuesta en el lado de la especificación, pero de forma mucho más ligera?

Una CLI. Tres comandos. Y su intención se transforma en una **especificación ejecutable** para el agente IA que ya utiliza.

---

## El problema que resuelve el GitHub spec-kit

Usted usa Cursor, Claude, Copilot, u otro agente IA para desarrollar. Le describe lo que quiere. Produce algo. No es exactamente eso, reformula, corrige, algo más cambia. Itera.

No es vibe coding en sentido estricto, tiene un objetivo. Pero el agente IA no tiene una spec. Trabaja desde un prompt, no desde un documento estructurado que describe con precisión qué debe hacer el sistema, cómo se mide el éxito, y qué no debe hacer bajo ningún concepto.

La diferencia entre un prompt y una spec es la diferencia entre "constrúyeme algo parecido a esto" y "este es el contrato exacto de lo que debes entregar".

El **GitHub spec-kit** produce ese pliego de condiciones estructurado. En tres comandos. Es lo que llamamos **spec-driven development** aplicado al agente IA: primero la especificación, luego el código.

---

## Los tres comandos del GitHub spec-kit

### `/specify`: transformar la intención en especificación agente IA

Usted describe su necesidad en lenguaje natural. `/specify` inicia un diálogo estructurado: plantea preguntas de clarificación, identifica las zonas grises y produce un documento de **especificación agente IA** completo.

Este documento contiene:
- El contexto y los objetivos
- Las funcionalidades requeridas, con su prioridad
- Los criterios de aceptación para cada funcionalidad
- Las restricciones técnicas y las dependencias
- Los casos límite y los comportamientos en caso de error

### `/plan`: descomponer la spec en pasos ejecutables

`/plan` traduce la spec en un plan de implementación para un **desarrollo IA estructurado**. Descompone el proyecto en módulos, secuencia las dependencias e identifica los puntos de atención técnicos.

Cada paso está delimitado, con un entregable esperado y criterios de validación. El avance se mide por lo que se ha entregado y validado, no por sensaciones subjetivas.

### `/tasks`: generar la lista de tareas para el agente

`/tasks` toma el plan y genera la lista de tareas concretas para el agente IA. Cada tarea es atómica, con su contexto, sus inputs, sus outputs esperados y sus dependencias.

Su agente IA ya no "adivina" qué debe hacer. Recibe una tarea delimitada, produce un output definido y pasa a la siguiente. Eso es una **especificación ejecutable** en acción.

Una hora para validar el documento. `/plan` secuenció el proyecto en seis módulos. `/tasks` generó 23 tareas para el agente.

## Los dos errores que debe evitar con el GitHub spec-kit

**Spec vaga, resultado vago.** Responda con precisión a las preguntas de `/specify`. "La herramienta debe gestionar los errores" no es una spec. "La herramienta muestra un mensaje de error explícito y reintenta automáticamente tras 30 segundos" sí lo es. Sin este nivel de detalle, la **especificación ejecutable** que genera el spec-kit no protege su proyecto.

**Valide la spec antes de codificar.** El documento producido por `/specify` es el punto de partida de una validación, no un paso a omitir. Léalo. Corrija lo que no corresponda. Ese momento de validación es lo que diferencia el **spec-driven development** del vibe coding con más pasos.
