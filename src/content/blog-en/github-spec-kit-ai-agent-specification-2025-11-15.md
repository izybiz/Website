# GitHub Spec-Kit: three commands so your AI agent builds what you actually want

Since May 2025, BMAD has demonstrated that vibe coding can be overcome. By imposing a team of specialised agents, analyst, PM, architect, developer, QA, each passing structured artefacts to the next, it guarantees that the AI produces what you actually specified. It works. It is also a full framework, with a real configuration curve.

One CLI. Three commands. Your intent becomes an executable specification the AI agent you are already using can follow faithfully. That is the principle of spec-driven development: fewer iterations, greater reliability, and a clear contract between what you want and what gets built.

---

## The problem the GitHub spec-kit solves

You use Cursor, Claude, Copilot, or another AI agent for development. You describe what you want. It produces something. It is not quite right, you rephrase, it corrects, something else changes. You iterate.

This is not vibe coding in the strict sense, you have an objective. But the AI agent has no spec. It is working from a prompt, not from a structured document that precisely describes what the system must do, how success is measured, and what must absolutely not happen.

The difference between a prompt and a spec is the difference between "build me something that looks like this" and "here is the exact contract of what you must deliver."

Spec-kit produces that structured brief. In three commands. This is the foundation of AI agent specification: constrain intent before the workflow opens, so the agent has nothing to guess.

---

## The three commands of the GitHub spec-kit

### `/specify`: turning intent into a structured document

You describe your requirement in plain language, what you want to build, for whom, in what context, under what constraints. `/specify` opens a structured dialogue: it asks clarifying questions, surfaces grey areas, and produces a complete AI agent specification document.

This document is not a requirements PDF. It is a structured specification file your AI agent can read as direct input. It contains:

- Context and objectives
- Required features, with their priority
- Acceptance criteria for each feature
- Technical constraints and dependencies
- Edge cases and error-handling behaviour

What `/specify` does that you will not do naturally in a prompt: it enforces exhaustiveness. It does not let you ignore edge cases. It does not let you write "the tool must handle errors" without defining what that means concretely. That is the role of executable specification: constrain intent before opening the workflow.

### `/plan`: breaking the spec into executable steps

Once the spec is produced, `/plan` translates it into an implementation plan. It decomposes the project into modules, sequences dependencies, and flags technical points of attention.

This plan is what your AI agent receives to work from. Not "build this module," but "here is the module, here is what it must do, here is what it depends on, here is how we validate it is correct."

The plan is also what allows you to steer. Each step is bounded, with an expected deliverable and validation criteria. You no longer measure progress by feel, you measure it by what has been delivered and what has been validated. This is structured AI development in practice: every task bounded, every output verifiable.

### `/tasks`: generating the task list for the agent

`/tasks` takes the plan and generates the concrete task list for the AI agent. Each task is atomic, with its context, inputs, expected outputs, and dependencies.

This is the level of instruction at which an AI agent works reliably. Not a general description, a precise task with a clear scope.

What this changes in practice: your AI agent no longer "guesses" what it should do. It receives a bounded task, produces a defined output, and moves to the next. The final result is coherent because each task was designed to be.

## The two mistakes to avoid with spec-kit

**A vague spec produces a vague result.** `/specify` asks questions, answer them precisely. "The tool must allow tracking of opportunities" is not a spec. "The tool displays open opportunities ranked by closing probability, with an alert when an opportunity has had no activity in the past 7 days" is.

**Validate the spec before coding.** The document produced by `/specify` is not final, it is the starting point for a validation. Read it. Correct what does not match. Add what is missing.
